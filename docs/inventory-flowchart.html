System Architecture Overview

The inventory system supports warehouse-scoped multi-tenancy with three stock locations

Warehouse Stock

Central inventory

Table: product_stock
Access: Admin/Manager
Scope: Per warehouse

Staff Stock

Field inventory

Table: staff_stock
Access: Self + Manager
Unique: user + product + warehouse

Movement Log

Audit trail

Table: stock_movements
Type: Immutable ledger
Records: All changes

Stock Flow Directions

WAREHOUSE
Central Stock
Agent #1
Staff Stock
→ W→S Transfer
← S→W Return
Agent #2
Staff Stock
→ W→S Transfer
⇄ S→S Transfer
Sales auto-deduct from: Staff Stock (first) or Warehouse (fallback)

Valid Operations

  • • Warehouse → Staff: Auto-approved, immediate
  • • Staff → Staff: Any staff with stock
  • • Sales: Deduct from staff first, warehouse fallback
  • • All operations are atomic (no partial updates)

Requires Approval

  • • Staff → Warehouse returns (Manager approval)
  • • Stock adjustments (Admin/Manager only)
  • • Negative stock handling

Stock Transfer Flow

Complete flow for warehouse-to-staff and staff-to-staff transfers

1

Warehouse → Staff Transfer

Auto-approved, immediate execution

UI

StockTransferModal.tsx

  • • User selects transfer type: "warehouse_to_staff"
  • • Selects source warehouse & destination staff
  • • Picks products and quantities
  • • Validates: quantity ≤ available stock
RPC

record_stock_transfer()

Validations:

  • • Check warehouse stock availability
  • • Verify user permissions
  • • Generate display_id (STF-YYYYMMDD-XXXX)

Creates: stock_transfers record with status='completed'

DB

execute_stock_transfer()

Atomic Operations:

  • 1. Deduct from product_stock (warehouse)
  • 2. UPSERT staff_stock (add quantity)
  • 3. Log to stock_movements

All in single transaction - rollback on any error

🔄

Query Invalidation

Invalidates: warehouse-stock, staff-stock, stock-movements, stock-transfers

2

Staff → Staff Transfer

Peer-to-peer transfer, no approval needed

👤
Source Staff
Must have stock
→
Direct Transfer
👤
Destination Staff
Receives stock

Note: Same warehouse required. Stock stays within the same warehouse context.

Sale Stock Deduction Flow

How recording a sale affects inventory

Phase 1: Pre-Sale Stock Check

check_stock_availability()

Called before showing sale confirmation

Admin/Manager → Checks warehouse stock
Agent/Marketer → Checks staff stock first, warehouse fallback
↓

Validation Result

  • ✓ Sufficient stock: Continue to sale
  • ✗ Insufficient: Block with error message
  • ✗ Check failed: Retry or offline mode

Phase 2: Atomic Sale Recording

All operations in single transaction
🔒
1. Lock Store Row
FOR UPDATE
📦
2. Deduct Stock
staff_stock or product_stock
📝
3. Create Sale
sales + sale_items
📊
4. Update Outstanding
store.outstanding

Stock Deduction Priority

1. Staff Stock (if has stock)
→
2. Warehouse Stock (fallback)
→
3. Block (if neither has stock)

Stock Return Flow

Staff returning stock to warehouse with manager approval

1

Staff Initiates Return

PENDING

Action

Staff creates transfer with type "staff_to_warehouse"

  • • Status set to 'pending'
  • • Stock NOT moved yet
  • • Manager notification sent

Data Created

stock_transfers {
  transfer_type: 'staff_to_warehouse',
  status: 'pending',
  quantity: requested_amount
}
2

Manager Reviews

REVIEW

review_stock_return() RPC

Actual Quantity

What was physically received

Difference

requested - actual

Action

'keep' or 'flag'

3

Resolution

✓ Approved

  • • Stock moved: Staff → Warehouse
  • • staff_stock quantity reduced
  • • product_stock quantity increased
  • • Logged to stock_movements

✗ Rejected

  • • Transfer marked 'rejected'
  • • No stock movement
  • • Staff keeps their stock
  • • Notification sent to staff

Variance Handling (if difference > 0)

Keep:

Difference remains with staff (write-off)

Flag:

Logged to staff_performance_logs for investigation

Current Implementation (Issues)

StockAdjustmentModal.tsx

// Direct table update!
await supabase
  .from('product_stock')
  .update({ quantity: newQty })
  .eq('id', stockId);

// Separate movement log
await supabase
  .from('stock_movements')
  .insert({...});
• Race Condition: Check and update are separate
• No Atomicity: Can leave partial data on failure
• Security: Bypasses RPC permission checks
✓

Recommended Implementation

New adjust_stock() RPC

CREATE FUNCTION adjust_stock(...)
RETURNS VOID
LANGUAGE plpgsql

BEGIN
  -- Permission check
  -- Validate warehouse scope
  -- Lock row FOR UPDATE
  -- Update stock
  -- Log movement
  -- Commit
END;
• Atomic: All operations in single transaction
• Safe: FOR UPDATE prevents race conditions
• Secure: Role validation in RPC
