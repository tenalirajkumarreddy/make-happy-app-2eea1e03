# Phone OTP Setup Guide

## ⚠️ CRITICAL SECURITY ISSUE - FIX FIRST

**Before enabling phone OTP, you MUST fix this security vulnerability:**

Your `public.otp_sessions` table has an RLS policy that allows **anonymous users unrestricted access** to all OTP sessions. This is a critical security risk.

**Run this SQL immediately:**
```sql
-- Remove insecure policy
DROP POLICY IF EXISTS "Anonymous users can manage OTP sessions" ON public.otp_sessions;

-- Add secure policies (adjust based on your auth flow)
CREATE POLICY "Users can create OTP sessions"
  ON public.otp_sessions FOR INSERT TO anon WITH CHECK (true);

CREATE POLICY "Users can read own OTP session"
  ON public.otp_sessions FOR SELECT TO anon
  USING (session_token = current_setting('request.jwt.claims', true)::json->>'session_token');
```

See `PHONE_OTP_CONTEXT.md` for full details.

## Overview
Phone OTP login has been re-enabled using Supabase's native phone authentication (no edge functions required for dev phase).

## Implementation

### Frontend Changes
- Updated `src/pages/Auth.tsx` to include phone OTP UI
- Added state variables: `phoneNumber`, `otpCode`, `otpSent`
- Implemented handlers: `handleSendOtp()` and `handleVerifyOtp()`
- Using Supabase's native `auth.signInWithOtp()` and `auth.verifyOtp()` methods

### How It Works
1. User enters phone number (with country code, e.g., +91XXXXXXXXXX or 10-digit)
2. App calls `supabase.auth.signInWithOtp({ phone: normalizedPhone })`
3. Supabase sends OTP via configured SMS provider
4. User enters OTP code
5. App calls `supabase.auth.verifyOtp({ phone, token, type: 'sms' })`
6. On success, user is authenticated and directed to registration or dashboard

## Supabase Configuration Required

### Step 1: Enable Phone Auth Provider
1. Go to Supabase Dashboard → Authentication → Providers
2. Enable "Phone" provider
3. Select an SMS provider (options below)

### Step 2: Configure SMS Provider

#### Option A: Twilio (Recommended for Production)
1. Sign up at https://www.twilio.com
2. Get your Account SID and Auth Token
3. Purchase a phone number
4. In Supabase Phone settings:
   - Provider: Twilio
   - Account SID: `your_account_sid`
   - Auth Token: `your_auth_token`
   - Twilio Phone Number: `+1234567890`

#### Option B: Twilio Verify (Alternative)
1. Same as Twilio but uses Verify service
2. Create a Verify Service in Twilio Console
3. Use Service SID instead of phone number

#### Option C: MessageBird
1. Sign up at https://www.messagebird.com
2. Get your API Key
3. In Supabase Phone settings:
   - Provider: MessageBird
   - API Key: `your_api_key`
   - Originator: Your sender ID

#### Option D: Vonage (Nexmo)
1. Sign up at https://www.vonage.com
2. Get your API Key and Secret
3. In Supabase Phone settings:
   - Provider: Vonage
   - API Key: `your_api_key`
   - API Secret: `your_api_secret`
   - From: Your sender number

#### Option E: Test Mode (Dev Only)
1. In Supabase Dashboard → Authentication → Settings
2. Enable "Enable phone confirmations"
3. For testing without real SMS:
   - OTP will be printed in Supabase logs (Dashboard → Logs)
   - OR use fixed test OTP in Settings

### Step 3: Configure OTP Settings (Optional)
1. Go to Authentication → Settings
2. Configure:
   - OTP expiry time (default: 60 seconds)
   - OTP length (default: 6 digits)
   - Rate limiting settings

### Step 4: Update RLS Policies (If Needed)
Ensure your RLS policies support phone-based authentication:
- Users created via phone will have `phone` field populated
- The `user_id` will be auto-generated by Supabase Auth

## Testing

### Without SMS Provider (Local Dev)
1. Check Supabase logs for OTP code after calling `signInWithOtp()`
2. Dashboard → Logs → filter for "phone" or "otp"
3. Copy the 6-digit code and use it in `verifyOtp()`

### With SMS Provider
1. Enter real phone number
2. Receive SMS with OTP
3. Enter OTP to verify

## Phone Number Format
- International format: `+<country_code><number>` (e.g., `+919876543210`)
- The app auto-normalizes 10-digit numbers by adding `+` prefix
- For Indian numbers: `+91` prefix is standard

## Migration from Firebase/OpenSMS
The previous implementation used:
- Firebase Phone Auth → ID Token → Edge Function → Supabase Session
- OR OpenSMS Gateway → Custom OTP → Edge Function → Supabase Session

New simplified flow:
- Direct Supabase Phone Auth (no Firebase, no custom edge functions)
- SMS sent via Supabase's configured provider
- Native Supabase session creation

## Rollback Instructions
If you need to revert to Google-only auth:
1. In `src/pages/Auth.tsx`, change the divider text back to "phone OTP disabled for now"
2. Replace the OTP form with the warning message
3. Set `const otpEnabled = false` flag
