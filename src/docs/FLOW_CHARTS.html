Key Rule - Sale vs Invoice:
✅ Sale: Auto-created when order is fulfilled (stock deducted, balance updated)
📝 Invoice: Optional - user decides whether to generate or not
📄 If invoice IS generated, it links to the Sale (not the Order)
💰 Payment is recorded against the Sale, invoice is just a document
⚠️ CRITICAL: Delivered Orders CANNOT Be Cancelled!

Once an order is marked DELIVERED:

  • ✅ Sale is auto-created (stock deducted, balance updated)
  • ❌ CANNOT use "Cancel" - Cancel only works for Pending/Confirmed orders
  • ↩️ Must use Sale Return to reverse the transaction
  • 📦 Sale Return will: Add stock back, clear outstanding, create audit trail

Why? Because the Sale has already affected inventory and financial records. A simple "cancel" would leave orphaned data. Sale Return properly reverses everything.

✅ Correct Actions by Status:
Status Can Cancel? Need to Reverse? Action
PENDING ✅ Yes ❌ No Cancel Order
CONFIRMED ✅ Yes ❌ No Cancel Order
DELIVERED ❌ NO ✅ Yes Sale Return
CANCELLED — — View only
Best Practice:
  • Always: Order → Fulfill → Sale created (automatic)
  • Optional: Generate Tax Invoice if customer needs a bill
  • Proforma: Generate before delivery only if customer requests a quote

Invoice Conversion Logic

Proforma → Tax Invoice Conversion:
  • When an order is delivered, the proforma invoice is NOT automatically converted
  • User manually creates a Tax Invoice from the delivered order
  • Proforma remains as historical reference (audit trail)
  • Tax Invoice becomes the official document for payment
  • Both invoices linked to the same order via order_ref
Best Practice:
  • Generate Proforma at Pending/Confirmed status (for customer approval)
  • Generate Tax Invoice at Delivered status (for payment)
  • Keep separate IDs for audit purposes
  • Link both to same order for traceability

📝 Key Takeaways

1. Proforma vs Tax Invoice - Keep Separate

Proforma is a "quote", Tax Invoice is the "bill". They serve different purposes and should have separate IDs.

2. Order-to-Sale Link

When an order is delivered, a sale record is created. The order is NOT deleted - it tracks the request history.

Linking:

  • orders.fulfilled_by_sale_id → Points to the auto-created sale
  • sales.fulfilled_order_id → Points back to the original order
  • 🔗 UI shows "View Sale" button on delivered orders for quick navigation
  • 📊 Both records are needed for complete audit trail
3. Invoice Cancellation Logic
  • Draft Invoice: Can be deleted (soft delete)
  • Issued Invoice: Can be cancelled (creates Credit Note if needed)
  • Cancelled Invoice: Can be hard deleted
  • If associated Sale exists, ask: "Delete invoice only or also reverse sale?"
4. Outstanding Balance Updates

Balance is updated when:

  • Sale is recorded (adds to outstanding)
  • Payment transaction recorded (reduces outstanding)
  • Credit Note issued (reduces outstanding)
5. Agent Field Delivery Workflow
  1. Marketer creates order & assigns to agent
  2. Agent sees assigned orders
  3. Agent delivers and marks order fulfilled
  4. Sale is auto-created (stock deducted, balance updated)
  5. Agent optionally generates Tax Invoice if customer needs bill
  6. Customer pays or signs credit acknowledgment
6. Sale vs Invoice - Critical Distinction
  • Sale: Auto-created on fulfillment (the transaction record)
  • Invoice: Optional document (for customer billing purposes)
  • Payment: Recorded against Sale, NOT Invoice
  • Outstanding: Calculated from Sales, not Invoices
  • Legal: Sale is the transaction, Invoice is just a document

⚠️ KEY RULE: Delivered Orders CANNOT Be Cancelled!

Once an order is marked DELIVERED, the "Cancel" button is removed. Why?

  • ✅ Sale was auto-created (stock deducted, balance updated)
  • ❌ Simple "Cancel" would leave orphaned sale record
  • ↩️ Must use Sale Return to properly reverse everything
  • 📦 Sale Return adds stock back, clears outstanding, creates audit trail

Cancel vs Sale Return - When to Use Each

Order Status Action Available Result Why?
PENDING ✅ Cancel Order → CANCELLED No sale created yet
CONFIRMED ✅ Cancel Order → CANCELLED No sale created yet
DELIVERED ❌ NO Cancel — Sale already created
DELIVERED ✅ Sale Return Stock + Balance restored Properly reverses sale

Step-by-Step Reversal Process:
  1. User realizes error: "This order was delivered by mistake!"
  2. Check if Sale exists: System confirms sale was auto-created
  3. Check payment status: Was any payment recorded?
  4. Create Sale Return:
    • Select products to return
    • Enter quantity to return
    • Add reason (e.g., "Delivered by mistake")
  5. System automatically:
    • Adds stock back to warehouse
    • Clears outstanding balance
    • Creates audit trail
  6. If payment recorded: Issue refund or credit note
  7. Order status: Changed to CANCELLED (if no payment) or stays DELIVERED with return note
✅ Best Practices
  • 🚫 NEVER delete orders, sales, or invoices - only reverse them
  • ✅ Always use Sale Return for delivered orders that need reversal
  • 📝 Add clear reason for every return (audit trail)
  • 👤 Only users with can_create_sale_returns permission can reverse
  • 📊 All reversals show in reports as "negative sales"
